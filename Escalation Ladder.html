Pattern · Local-first chat with selective escalation

Asking Claude Code for help.

A local model in Ollama handles ordinary chat on your machine. When a turn needs to read files, run commands, or reason across a codebase, it hands off — quietly, in the background — to Claude Code, then stitches the answer back into the conversation.

BRAIN  2× Orange Pi 5 Pro · RK3588S

§ 00 The three actors

AWho's in the room.

One chat window, three minds behind it. The user never has to know where any given answer came from — that's the contract.

  • User — types in a normal chat box, expects normal turn‑taking.
  • Ollama — fast, cheap, on‑device. Owns the conversation: tone, memory, formatting, the decision of when to call for help.
  • Claude Code — runs headless in the background. Has shell, filesystem, and stronger reasoning. Returns a transcript or a structured result.
Ollama is the host. Claude Code is the specialist on call. The user only ever talks to the host.

§ 02 The escalation decision

CHow Ollama knows it's out of its depth.

Two strategies, used together in practice. The cheap one runs first; the expensive one — Ollama itself — only fires if the heuristic is ambiguous.

STRATEGY A cheap · ~0 ms

Heuristic prefilter

Regex / keyword pass on the incoming message. Fast, dumb, conservative — biased toward escalating, because escalating something easy is cheaper than missing something hard.

router/heuristic.pydef needs_claude(msg, ctx):
    # obvious tells
    if ctx.attached_files: return True
    if re.search(r"\b(refactor|rewrite|debug|test|run|grep)\b", msg, re.I):
        return True
    if re.search(r"\.(py|ts|tsx|go|rs|java|sql)\b", msg):
        return True
    if re.search(r"[/~]\S+/", msg):  # paths
        return True
    return None  # inconclusive → ask the model
STRATEGY B model-judged · 1 token

Expose escalate_to_claude_code as a tool. Ollama either answers itself, or emits a tool call whose arguments are the handoff payload. No separate classifier needed.

§ 03 What gets sent across

DThe handoff payload, and the prompt it becomes.

Whatever the transport mechanism, the contents of the handoff are the same shape. Treat it as a small contract between the two models: structured on the wire, prose by the time it hits Claude Code.

Payload (the wire format)

What Ollama assembles before calling out. Five fields, in priority order.

handoff.json{
  "goal": "Add rate limiting to the /login route
           and write a test that proves it triggers
           after 5 attempts in 60s.",

  "workdir": "/Users/me/code/acme-api",

  "conversation_summary": "User is debugging
     brute-force logins seen in prod. Stack is
     Express + Redis. They tried a naive in-memory
     counter; want something Redis-backed.",

  "attached_files": [
    "src/routes/login.ts",
    "src/middleware/"
  ],

  "acceptance": [
    "new test passes locally",
    "no existing tests regress",
    "diff is < 80 lines"
  ]
}

Prompt (what Claude Code actually reads)

Render the JSON into a brief, plain‑English brief. Don't dump the JSON itself.

prompt.md# Task

§ 04 Four ways to actually make the call

§ 05 Stitching the answer back into the chat

FDon't make the user stare at a blank cursor.

Claude Code's stream-json output emits events as it works. Ollama's job is to translate those events into single‑sentence updates in the user's voice — not to forward them raw.

Raw stream from Claude Code

stdout · ndjson{"type":"tool_use","name":"Read",
 "path":"src/routes/login.ts"}
{"type":"tool_use","name":"Grep",
 "pattern":"rate.?limit"}
{"type":"tool_use","name":"Edit",
 "path":"src/middleware/rateLimit.ts"}
{"type":"tool_use","name":"Bash",
 "cmd":"npm test -- login"}
{"type":"assistant_text",
 "text":"All 14 tests pass. Diff below."}

What the user sees in Ollama's chat

chat surface · user view› Reading login.ts and
  searching for existing rate limiters…

› Drafting a new middleware
  src/middleware/rateLimit.ts…

› Running the login test suite…

  All 14 tests pass.
  Here's the diff:

  + import rateLimit from "./middleware/...
  + app.use("/login", rateLimit({ ... }))
  ...

Translation pass

Each event becomes a single line, written by Ollama in the user's voice. The system prompt for this is two sentences: “You are narrating a colleague's work. Use present continuous tense. Never quote raw tool names.”

Backpressure

Throttle to one update per ~800ms. Claude Code may emit dozens of events per second on a grep; the user only needs to know something is happening.

Final stitch

When the stream closes, Ollama gets one more turn: re‑read the conversation, the user's last message, and Claude Code's final text — then write the closing answer itself. This is where tone and memory live.

§ 08 Which model on which box

IPick for the routing, not the coding.

Counter‑intuitive, but central: in this pattern the local model never has to be a great programmer — Claude Code is. The local model's real job is to recognise a coding task, emit the escalation tool call cleanly, and write a faithful §03 brief. So rank candidates in this order:

  • Tool‑calling reliability — does it emit valid function calls every time? RKLLama lists Qwen and Llama‑3.2+ as the formats with complete tool‑call support, which all but decides this.
  • Instruction following — can it summarise the conversation into a clean brief and know when not to escalate?
  • Coding literacy — only enough to handle one‑line fixes and write a sensible brief. The hard edits leave the network.

§ 09 The honest accounting

JIs this rig actually worth it?

Straight answer: for the goal of shipping code with Claude Code, the three boxes are not of significant practical benefit over just pointing the claude CLI at your repo. The reason is structural — the local tier can only absorb the work it's capable of, which is exactly the trivial turns it routes away from. All the cost, latency, and privacy exposure live in the hard turns, and those escalate regardless.

What it genuinely buys you

  • Learning & homelab valueThe strongest real return — understanding edge inference, NPUs, and routing. But it's personal, not economic.
  • An ambient front-endA 1.5 W device that's always on the desk is a different object than opening a laptop. A genuine, if modest, UX win.
  • Offline reflexesTrivial Q&A and status keep working when the network drops — useful for a kiosk or air-gapped triage.
  • Privacy — partialOnly for turns that never escalate. Real, but narrow.

What it doesn't — for a coding goal

  • Token savingsThe hard turns escalate anyway; the chatter you deflect is worth pennies. You'd never amortise ~$260+ of boards against it.
  • Privacy on the actual workThe moment a coding task escalates, your brief and code leave the network — identical to using Claude directly.
  • SpeedA 3–7 t/s router isn't doing the heavy lifting; it's a gate in front of the thing that is.
  • A Claude replacementA 0.5–7B can't code at that level. That's the whole reason it escalates.
Treat the current rig as a tinkering and ambient-interface project that can reach Claude Code — not a way to make Claude cheaper or more private. If you want the local tier to actually finish real coding work before escalating, that stops being a software question and becomes a hardware one. ↓

Does your M5Stack_I2C_Framework reduce complexity?

verified · source read

Yes — it removes the single most fiddly chunk outright, just not the one its name implies. It's a board-aware (CoreS3 / Core2 / Tough / Core1, auto-detected) plugin framework that exposes M5Stack devices over an HTTPS REST API + live dashboard (also MQTT / SD / serial / LCD). And it already ships UartDevice_ModuleLLM.h: a complete, debugged integration of the AX630C Module LLM.

What it already solves — the exact embedded landmines from the table above:

  • StackFlow JSON-over-UART via the official M5Module-LLM lib — async token streaming, never stalls the loop.
  • The nastiest AX630C bug — StackFlow unit-leak across host reboots — handled with sys.reset() + RX-buffer tuning.
  • The LLM exposed as GET /api/llm/set?ask=… + /api/llm, with a dashboard chat panel, TLS, and auth.

What it doesn't — and reshapes

It makes the CoreS3 a networked Module-LLM micro-service (any device can hit /api/llm) — not a terminal that forwards. There's no escalation tool and no call out to the Pi or Claude.

So the routing glue lives outside it: either an orchestrator that calls /api/llm for trivial turns and Claude Code for hard ones, or a new forwarding plugin you add — clean, since you own the plugin model.

Net effect: the M5 firmware rung drops from Moderate · 1–3 days to Easy · hours — the hard-won part is done and battle-tested. But total project effort still barely moves: the Orange Pi NPU + router remains the bottleneck, and this framework doesn't touch it. Its real gift is a clean, networked, extensible base you already control — the ideal place to bolt the escalation forwarder onto.

§ 12 The forwarding plugin

MTwo ways to wire the router.

The “router + voice” tier from §00–§05 can live in one of two places. Option A puts it on the Core as a drop-in framework plugin. Option B puts it in an external orchestrator and leaves the Core firmware untouched. Same job, different home — pick by whether you want the Core self-contained or the routing easy to iterate.

Option A · on the Core — NetDevice_Router.h

Makes the CoreS3 the single chat entry point: it classifies each turn on-device, answers trivial ones from the on-board Module LLM, and escalates hard ones to the Pi — which fronts Claude Code.

No new base class, no edits to Framework. IPinDevice already provides the non-I2C “registered = active via beginPins()” path — the router just subclasses it, exactly as IUartDevice does.

The decoupling trick

For trivial turns it delegates to the Module LLM and then mirrors its streamed answer through the public toJson() — so it reads the reply with zero coupling to the other plugin's internals. The LLM's own fastPoll() drives the UART; the router only observes.

The escalated path expects the Pi to answer text/event-stream — lines of data: {"delta":"…"} ending in data: [DONE]. That's the one seam to match to your orchestrator; it lives entirely in _consumeLine(). Everything else — classify, delegate, mirror, time-out, dashboard, logging — is done.

Option B · external orchestrator — orchestrator.py

The same routing, moved off the Core into a small Python service (runs on the Pi or any box). The Core needs zero new firmware — the /api/llm endpoint that UartDevice_ModuleLLM.h already exposes is the trivial path. And it's mostly the Pi orchestrator you were going to write anyway, with one branch that calls the Core instead of Claude.

Lower effort than Option A: ~150–200 lines of plain Python, no MCU async, no reflash to change routing. You iterate the classifier in seconds.

The shape of it

Three pieces: a Core client (fire-and-poll /api/llm), a one-line classifier, and a streamed shell-out to claude -p. A handle() generator ties them together and keeps a short history for the brief.

§ 13 Can the Core call Claude itself?

NYes — but mind the word “Code.”

The CoreS3 can reach the cloud on its own — it has WiFi and TLS already. But “ask Claude” splits into two very different things, and only one of them fits on an ESP32.

✓ Fits on the Core

Claude — the API

Text in, text out. A chatbot reply: “explain this trace,” “draft this regex,” “summarise this.” The Core POSTs to api.anthropic.com/v1/messages and streams the answer back. ~80-line plugin.

✗ Cannot run on the Core

Claude Code — the agent

Needs a filesystem, shell, git repo, and Node to run its read→edit→test loop. The ESP32 has none of these. It can trigger Claude Code on the Pi — never be it.

A direct call gets you a worded answer, not a code change with a diff and passing tests — which was the entire reason hard turns escalate in the first place. For anything that touches a codebase, Claude Code still has to run where the repo lives.

Even for pure text, three reasons to think twice

01 · KEY EXPOSURE

The API key sits in firmware on a desk gadget whose flash can be read. The Pi keeps it on a real machine you control.

02 · NO TOOLS

Exactly what you escalate for — repo edits, running tests — is what a direct call can't do.

03 · GUTS THE DESIGN

If the Core just phones the cloud, the local NPU tier is pointless — you've built a thin cloud client with a screen.

The tasteful use: a three-way router

If you do add it, don't make it the default — make it the middle of three routes. Trivial stays local; “smart text with no repo” goes to the Claude API; genuine coding tasks still escalate to Claude Code on the Pi. Cheap things stay cheap, you get a strong cloud answer when no codebase is involved, and the full agent only spins up when there's actually something to edit.

The direct-API branch ships as NetDevice_ClaudeAPI.h — a sibling to NetDevice_Router.h, same IPinDevice base, same streaming pattern. Register it and add one branch to the router's classifier.

§ 14 Two deployment shapes

OCore alone, or Core with a Pi.

The same CoreS3 firmware supports two very different topologies, decided entirely by which plugins you register. Pick by one question: do you need work that touches a real codebase? If never, the Core stands alone. If yes, a Pi (or any always-on box) joins as the agent tier.

A · Core alone — no Pi

two tiers

The CoreS3 is the whole device. It answers trivial turns from its on-board Module LLM, and — optionally — sends “smart text” turns straight to the Claude API over the internet. No second machine. There is no Claude Code here: with no filesystem or shell, the Core can get worded answers but never a code change with a diff.

Fig. O-A  ·  Core alone. Plugins: UartDevice_ModuleLLM + optional NetDevice_ClaudeAPI.

Good for

A self-contained desk assistant: Q&A, drafting, explaining. Works with the network down for local turns.

Cost / setup

One device. No server to run or maintain. Flash and go.

The ceiling

Answers only. No file edits, no tests, no diffs — and the API key lives in firmware.

B · Core with a Pi — the full ladder

three tiers

The Core becomes the front-end and router; a Pi (or any always-on Linux box) joins as the brain + agent host. Trivial turns stay on the Core, hard ones escalate to an orchestrator on the Pi, which runs Claude Code against a real git checkout and streams a summary + diff back. This is the only shape that finishes coding work.

Fig. O-B  ·  Core + Pi. Plugins: UartDevice_ModuleLLM + NetDevice_Router → orchestrator.py → Claude Code.

Good for

Real coding: refactors, tests, debugging that produce a diff. Cheap turns stay local; the key stays on the Pi.

Cost / setup

A second always-on box plus the orchestrator. The NPU tier is where the real build effort lives (§11).

The payoff

Only the genuinely hard part ever reaches the cloud — the agent runs where your code is.

Same firmware, same Config.h. Topology A is ModuleLLM (+ optional ClaudeAPI); Topology B adds NetDevice_Router pointed at a Pi. You can start with A on the bench and grow into B by registering one more plugin and standing up the orchestrator — nothing else changes.
§ 15 Running A and B together

POne router, three destinations.

The two topologies aren't exclusive — kept live at once, they make the Core a three-way router. The classifier picks per turn: trivial stays local, “smart text” with no repo takes one hop to the Claude API, and genuine coding escalates to Claude Code on the Pi. Cheap stays cheap; the heavy agent only wakes for work that produces a diff.

Fig. P  ·  _classify() → local | direct_api | escalated, each streamed back the same way.

Two cases that earn the third route

  • Latency & cost. A one-hop API answer beats booting the full Claude Code agent — tool loop, repo read, test run — for a question that yields no diff.
  • Resilience. With the Pi powered down, smart-text answers keep working over the API; the device isn't reduced to just the 0.5B. The local tier still covers offline.

Each path also backstops a different failure: lose the internet and local still answers; lose the Pi and the API still answers.

The cost you accept

The third route means CLAUDE_API_KEY now lives in the Core's flash — a second key surface beside the Pi's. Worth it only if surviving a Pi outage matters to you.

If it doesn't, keep ROUTER_DIRECT_API off and let the Pi orchestrator own every cloud call — the Core then holds no key at all. That's the safer default; the third route is opt-in.

Two ways to place the three routes

Preferred · key stays on the Pi

The Core only ever calls the Pi. The orchestrator does the three-way itself — local-via-Core, direct-API, or Claude Code. The Core holds no key.

Simplest and safest. The trade: the direct-API path dies when the Pi is off.

Resilience · key on the Core

Register both NetDevice_ClaudeAPI and NetDevice_Router on the Core; the router gains a third branch that calls the API plugin directly.

Survives a Pi outage — at the cost of the key in flash. This is “A + B truly together on the device.”
